# 2. 模块划分

## 2.1 总览

采用 **Maven 多模块单体架构**——编译期模块隔离，运行期单进程部署。不做微服务拆分，理由：

- 秒杀核心链路（限流→校验→扣库存→下单）是低延迟敏感路径，跨进程 RPC 的序列化与网络开销不可接受
- 双机物理环境（Node A 单机 16GB），微服务进程数过多反而争抢 CPU
- 面试场景下，清晰的模块边界 + 单进程内的事件驱动 比徒有其表的微服务更能展示架构能力

```
seckill-flash-system (parent)
  ├── seckill-common        # 公共基础设施
  ├── seckill-product       # 商品与活动管理
  ├── seckill-gateway       # 网关层 Filter Chain
  ├── seckill-engine        # 秒杀核心引擎
  ├── seckill-order         # 订单中心
  ├── seckill-mq            # 消息队列基础设施（Topic 常量 + RocketMQTemplate）
  └── seckill-web           # Spring Boot 启动入口，聚合所有模块
```

## 2.2 模块详情

### seckill-common（公共基础设施）

| 项 | 内容 |
|---|------|
| **职责** | 所有模块共享的类型、工具、配置，零业务逻辑 |
| **包含** | 统一返回码枚举 `ResultCode`、全局异常 `BizException`、通用响应体 `ApiResponse<T>`、Redis Key 命名常量、Hash Tag 工具方法、JSON 序列化工具 |
| **依赖** | 无内部模块依赖 |
| **外部依赖** | spring-boot-starter-data-redis、jackson、lombok |

### seckill-product（商品与活动管理）

| 项 | 内容 |
|---|------|
| **职责** | 秒杀活动 CRUD、商品信息查询、**库存预热至 Redis** |
| **包含** | `SeckillActivity` 实体、`Product` 实体、活动配置 Mapper、库存预热 Service（启动时将 MySQL `total_stock` 写入 Redis 单 Key）、活动状态管理（未开始/进行中/已结束） |
| **依赖** | seckill-common |
| **外部依赖** | mybatis-plus、mysql-connector |

### seckill-gateway（网关层 Filter Chain）

| 项 | 内容 |
|---|------|
| **职责** | **请求的第一道防线**——在请求到达秒杀引擎前完成拦截。不包含秒杀业务逻辑 |
| **包含** | `IpRateLimitFilter`（IP 级令牌桶，Caffeine `maximumSize(10000)` 有界缓存 + Guava RateLimiter，不碰 Redis，满时驱逐放行至下层）、`UserBloomFilter`（Bloom Filter 快速否决不存在的 userId）、`RateLimitFilter`（userId 滑动窗口限流，Redis ZSet）、`WaterLevelFilter`（QPS 水位检测 → 触发验证码降级）、`DynamicUrlFilter`（Token 校验） |
| **依赖** | seckill-common |
| **外部依赖** | spring-boot-starter-web（Filter 注册）、spring-boot-starter-data-redis、guava、caffeine |

**Filter Chain 执行顺序（不可颠倒——见 ADR-009）**：
```
请求 → IpRateLimitFilter → UserBloomFilter → RateLimitFilter → WaterLevelFilter → DynamicUrlFilter → Controller → seckill-engine
       (IP 令牌桶)         (Bloom 否决)      (userId 滑动窗口)  (水位检测/降级)    (Token 校验)

关键设计: IP 和 Bloom 两道闸不碰 Redis，在本地内存中完成（纳秒/微秒级）
         → 恶意请求在进 Redis 之前就被否决
         → 只有合法用户的请求才有资格在 Redis 中创建限流 Key
```

| 项 | 内容 |

| **职责** | **秒杀系统的灵魂**——执行 Lua 脚本，原子完成库存扣减与购买记录写入，返回确定性结果 |

### seckill-order（订单中心——拥有完整的订单生命周期）

| 项 | 内容 |
|---|------|
| **职责** | 订单的**完整生命周期管理**：通过 RocketMQ 事务消息（Two-Phase Commit）原子绑定订单创建与延时消息投递、状态机驱动、超时订单消费与库存回池。**Consumer 监听器在此模块内部，不散落在外** |
| **包含** | `Order` 实体、`OrderStatus` 枚举（PENDING_PAY / PAID / CANCELLED / TIMEOUT）、`OrderService`（创建订单 + RocketMQ 事务消息执行）、`OrderTransactionListener`（RocketMQ TransactionListener 实现——executeLocalTransaction + checkLocalTransaction）、`OrderTimeoutConsumer`（RocketMQ Consumer 监听器，核验支付状态 → INCRBY Redis 回池 → 更新订单状态） |
| **依赖** | seckill-common、seckill-mq（仅使用其 `MqConstants` 和 `RocketMQTemplate`） |
| **外部依赖** | mybatis-plus、mysql-connector、rocketmq-spring-boot-starter、**redisson-spring-boot-starter** |

**订单状态机**：
```
                  seckill-engine 返回 SUCCESS
                         │
                         ▼
                  PENDING_PAY（待支付）
                    │           │
           用户支付  │           │ MQ Consumer 收到延时消息（5分钟到期）
                    ▼           ▼
                  PAID      CANCELLED
                （已完成）   （已取消 → INCRBY Redis 库存回池）
```

**Redisson 分布式锁——订单防并发脏写**：
```
为什么需要 Redisson 而非 Jedis SETNX:
  场景: MQ Consumer 正要 UPDATE order SET status=3 时，用户刚好完成支付
  两个线程同时操作同一行 MySQL 数据 → 状态覆盖 → 已支付订单被误标为超时取消

  Redisson RLock 防护:
    lock = redisson.getLock("order:lock:{orderId}")
    lock.lock(10, TimeUnit.SECONDS)  // Watchdog 自动续期，对抗 GC 停顿
    try {
        order = mapper.selectById(orderId)
        if (order.status == PENDING_PAY) {
            // Consumer: INCRBY 回池 + UPDATE status=3
        } else if (order.status == PAID) {
            // Consumer: ACK，什么都不做（用户已支付）
        }
    } finally {
        lock.unlock()
    }

  Watchdog 机制: 默认每 10 秒续期一次，直到 unlock() 显式释放
  → 杜绝"锁在 GC 期间过期导致并发冲突"的经典生产事故

  Redis 宕机降级: 见 5.2 节——catch(RedisException) → 跳过 Redisson → SELECT FOR UPDATE 兜底
```

**RocketMQ 事务消息（Two-Phase Commit）——零表零轮询的最终一致性**：
```
为什么不用 Transactional Outbox:
  Outbox 表 + OutboxScheduler 在高并发下是性能自毁:
    1万订单/s → 1万行 INSERT outbox + 每秒 SELECT * WHERE status=0 LIMIT 100
    → InnoDB 聚簇索引页分裂 + 行锁/间隙锁竞争
    → MySQL IOPS 被打满 → 整个订单入库链路卡死 ❌

  RocketMQ 是阿里出品，事务消息是其最核心的杀手锏特性。
  放着火箭不用，手搓独轮车，是对技术栈选择的根本性错误。

RocketMQ 事务消息三阶段——完美替代 Outbox:

  ┌─────────────────────────────────────────────────────────┐
  │ Phase 0: 预生成订单 ID（Snowflake）                     │
  │   Long orderId = snowflake.nextId();  // 必须在发半消息前生成 │
  │                                                         │
  │   □□□ 不能依赖 AUTO_INCREMENT——订单 ID 需要在 Half Message   │
  │   中作为消息体传递给 Consumer，否则 Consumer 无法关联订单 □□□│
  │                                                         │
  │   Snowflake 结构:                                       │
  │     [41bit 时间戳][10bit 机器ID][12bit 序列号] = 64bit   │
  │     单机 QPS ≈ 400 万/秒, 双机无冲突 ✓                  │
  ├─────────────────────────────────────────────────────────┤
  │ Phase 1: Send Half Message (含预生成的 orderId)         │
  │   OrderTimeoutMessage body = {orderId, goodsId}             │
  │   rocketMQTemplate.sendMessageInTransaction(            │
  │       txProducerGroup, topic,                          │
  │       body,                     // Half Message         │
  │       orderId                   // 业务参数 → executeLocalTransaction 的 arg │
  │   )                                                     │
  │   → MQ Broker 收到，持久化，标记为"半消息"              │
  │   → 下游 Consumer 不可见（消息未提交）                  │
  ├─────────────────────────────────────────────────────────┤
  │ Phase 2: executeLocalTransaction (MQ 回调)              │
  │   public LocalTransactionState executeLocalTransaction( │
  │       Message msg, Object arg) {                        │
  │                                                         │
  │       Long orderId = (Long) arg;                        │
  │       try {                                             │
  │           INSERT INTO seckill_order (id, ...);  // 仅写订单，无 outbox 表│
  │           return LocalTransactionState.COMMIT_MESSAGE;   │
  │       } catch (Exception e) {                           │
  │           return LocalTransactionState.ROLLBACK_MESSAGE; │
  │       }                                                 │
  │   }                                                     │
  │                                                         │
  │   COMMIT → Broker 将半消息标记为可投递 → 5分钟延时生效  │
  │   ROLLBACK → Broker 丢弃半消息 → 无延时消息，无泄漏     │
  ├─────────────────────────────────────────────────────────┤
  │ Phase 3: checkLocalTransaction (异常兜底)               │
  │   public LocalTransactionState checkLocalTransaction(   │
  │       MessageExt msg) {                                 │
  │                                                         │
  │       // JVM 在 COMMIT/ROLLBACK 返回前宕机时            │
  │       // Broker 会主动回调此方法                         │
  │       Long orderId = parseOrderId(msg);                 │
  │       Order order = mapper.selectById(orderId);         │
  │                                                         │
  │       if (order != null) {                              │
  │           return LocalTransactionState.COMMIT_MESSAGE;   │
  │       } else {                                         │
  │           return LocalTransactionState.ROLLBACK_MESSAGE; │
  │       }                                                 │
  │   }                                                     │
  └─────────────────────────────────────────────────────────┘

架构收益:
  1. 零 MySQL 轮询 —— 轮询负担转移给 RocketMQ Broker（性能强悍的 C++ 实现）
  2. 零 outbox 表 —— 不需要 seckill_outbox 表，不需要 OutboxScheduler 线程
  3. JVM 宕机兜底 —— checkLocalTransaction 主动回查，最终一致性闭环
  4. Consumer 幂等天然可复用 —— 延时消息到达后查 order.status，PAID/CANCELLED 则 ACK
```

## 5.1 TCP 连接层物理熔断

### 故障场景

秒杀核心链路依赖三条 TCP 连接：`客户端 → Undertow`、`Spring Boot → Redis`、`Spring Boot → MySQL`。当 10 万 QPS 瞬间涌向单台机器，真正的危机不在 JVM 内部——在操作系统还没把连接交给 Java 进程之前，TCP 连接就可能已经死在内核队列里了。

### 真正的四层物理熔断链（从网卡到业务代码）

```
                                ┌──────────────────────────────────────────┐
第零层: OS 内核 TCP 协议栈       │ 客户端 ──SYN──> 操作系统                   │
(物理熔断的起点)                │                                            │
                                │ net.core.somaxconn = 65535                │
                                │   → accept 队列（ESTABLISHED 但未被 accept）│
                                │   默认 128 → 10 万并发下瞬间溢出           │
                                │   → 溢出时内核行为取决于 tcp_abort_on_overflow │
                                │                                            │
                                │ net.ipv4.tcp_max_syn_backlog = 65535       │
                                │   → SYN 队列（收到 SYN 但未完成三次握手）    │
                                │   默认 128~1024 → 溢出时直接丢包（客户端感知 │
                                │   为 SYN 超时，60~120 秒后才报错）          │
                                │                                            │
                                │ net.ipv4.tcp_abort_on_overflow = 1 □□□     │
                                │   → 默认 0: 队列满时内核静默丢包            │
                                │   → 设为 1: 队列满时内核直接回 RST 包       │
                                │   → 客户端立刻收到 "Connection refused"     │
                                │   → 这是真正的"TCP 层物理熔断"             │
                                │                                            │
                                │ net.ipv4.tcp_tw_reuse = 1                  │
                                │   → 复用 TIME_WAIT 端口, 缓解端口耗尽       │
                                │ net.ipv4.ip_local_port_range = 1024 65535   │
                                │   → 增大临时端口范围 (SpringBoot→Redis 用) │
                                └──────────────────────────────────────────┘
                                ┌──────────────────────────────────────────┐
第一层: Undertow 容器             │ server.undertow.io-threads = 12           │
(将连接拉入 JVM)                │   → IO 线程数 = CPU 核心数                │
                                │ server.undertow.worker-threads = 64        │
                                │   → 工作线程 = IO × 5~6                   │
                                │ server.undertow.buffer-size = 16384        │
                                │   → 16KB 缓冲区（匹配常见请求大小）         │
                                │                                            │
                                │ Undertow 的 accept backlog 继承自 OS       │
                                │ somaxconn → 65535 → 单机可同时 accept      │
                                │ 大量连接, Kernel 层不再丢包                │
                                └──────────────────────────────────────────┘
                                ┌──────────────────────────────────────────┐
第二层: 中间件连接池              │ Redis Lettuce:                            │
(连接管理)                      │   max-active=50, max-wait=50ms             │
                                │   timeout=200ms                           │
                                │ RocketMQ Producer:                        │
                                │   sendMsgTimeout=500ms                    │
                                │ MySQL HikariCP:                            │
                                │   maximumPoolSize=20                       │
                                │   connectionTimeout=1000ms                 │
                                └──────────────────────────────────────────┘
                                ┌──────────────────────────────────────────┐
第三层: 熔断器 + 应用降级         │ Sentinel 慢调用比 > 20% → OPEN (10s)      │
(应用层决策)                    │ catch(RedisConnectionFailureException)    │
                                │   → SeckillResult.SYSTEM_BUSY             │
                                └──────────────────────────────────────────┘
```

### 为什么"连接池超时"不能叫 TCP 熔断

```
默认 Linux TCP 行为（somaxconn=128, tcp_abort_on_overflow=0）:
  客户端 SYN → 服务器内核
    ├── accept 队列有空位 → 完成握手 → 交给 Undertow → OK
    └── accept 队列已满(128) → 内核静默丢包
         → 客户端 TCP 栈重传 SYN（指数退避: 1s, 2s, 4s, 8s...）
         → 最终超时(60~120s) → 客户端报错
         → 整个过程 JVM 毫不知情——没有任何 Java 代码被执行
         → Sentinel、try-catch、连接池超时——全部形同虚设 ❌

修正后 Linux TCP 行为（somaxconn=65535, tcp_abort_on_overflow=1）:
  客户端 SYN → 服务器内核
    ├── accept 队列有空位 → 正常连接 ✓
    └── accept 队列已满(65535) → 内核直接回 RST
         → 客户端立刻收到 "Connection reset by peer"（< 1ms）
         → 客户端可立即重试或降级
         → 这才是"物理层熔断"——操作系统而非应用层做判决
```

### sysctl 完整配置脚本

```bash
# /etc/sysctl.d/99-seckill.conf
# 在双机环境中，两台机器均需执行

# ── TCP 全连接队列（accept 队列）──
net.core.somaxconn = 65535

# ── TCP 半连接队列（SYN 队列）──
net.ipv4.tcp_max_syn_backlog = 65535

# ── 队列溢出时的行为: 直接回 RST，不静默丢包 ──
net.ipv4.tcp_abort_on_overflow = 1

# ── TIME_WAIT 端口复用（减轻 SpringBoot→Redis 连接压力）──
net.ipv4.tcp_tw_reuse = 1

# ── 本地临时端口范围（扩大可用端口数）──
net.ipv4.ip_local_port_range = 1024 65535

# 生效: sysctl -p /etc/sysctl.d/99-seckill.conf
```

### 面试口述要点

> "很多人说'TCP 熔断'只想到连接池超时，但真正的 TCP 熔断发生在操作系统内核。默认 Linux 的 somaxconn 只有 128——10 万并发下，超过 128 个连接后内核直接静默丢包，客户端 SYN 重传等 60 秒才超时。整个过程没有任何 Java 代码被执行，你的 Sentinel 和 try-catch 全部形同虚设。我的修正很简单：somaxconn 拉到 65535，打开 tcp_abort_on_overflow，溢出时内核直接回 RST——让客户端在 1 毫秒内就知道'此路不通'。这才是物理层的真正短路。"

## 5.2 异步补偿期"重试风暴"与防超卖

### 故障场景（时间线推演）

```
T+0s:  开闸，100 人秒杀成功，100 条订单写入 MySQL，100 条事务消息在 MQ 中倒计时
T+5min: 100 条消息同时到期，全部推送给 OrderTimeoutConsumer

假设此刻 MySQL 因磁盘 IO 抖动（或 Node B 网络拥塞）:
  1. OrderTimeoutConsumer 线程池 (16 线程) 同时拉取 100 条消息
  2. 每条消息尝试获取 Redisson RLock → 因网络抖动，lock.tryLock(100ms) 大部分超时
  3. 获取锁失败 → 代码抛出异常 → RocketMQ 触发消息重试
  4. 100 条消息按重试策略重新入队 (delayLevel=10s)
  5. 10s 后 100 条重试消息 + 新到的其他超时消息同时涌来
  6. 重试消息中的锁竞争 + DB 压力 → 更多失败 → 更多重试 → 指数级放大
  7. "重试风暴"成形 → 线程池耗尽 → 所有消费停滞 → 库存物理泄露
```

### 防御体系（四层）

```
第一层: Consumer 消费限流（源头扼杀）
  RocketMQ Consumer 配置:
    consumeThreadMin = 2      // 最小 2 线程（非洪峰时段省资源）
    consumeThreadMax = 4      // 最大 4 线程（刻意压低，不让消费抢 Redis/DB 资源）
    pullBatchSize    = 8      // 每次拉取最多 8 条
    // → 峰值 QPS: 4 线程 × (1s / avg 100ms per msg) ≈ 40 msg/s，温和可控
    // 注意: 超时订单处理晚几秒不是事故，系统被打垮才是事故

第二层 + 第三层(合并): Redisson 锁获取 + MySQL 降级——物理可达的降级路径

  □□□ 致命陷阱: tryLock 失败 → return RECONSUME_LATER → MySQL 降级代码永不可达 □□□
  PPT 架构矛盾:
    第二层: lock.tryLock(100ms) 失败 → return RECONSUME_LATER (方法结束)
    第三层: "即便 Redisson 不可用，SELECT FOR UPDATE 仍然兜底"
    物理真相: 方法在第二层就 return 了，第三层的代码永远不会被执行 ❌

  区分两类 tryLock 失败——必须用 try-catch 显式分流:
    
    boolean redisFailed = false;
    RLock lock = redisson.getLock("order:lock:" + orderId);
    
    try {
        if (lock.tryLock(100, TimeUnit.MILLISECONDS)) {
            // 正常路径: Redisson 锁获取成功
            try {
                processTimeout(orderId);  // 内部含 SELECT FOR UPDATE 双重保护
            } finally {
                lock.unlock();
            }
            return CONSUME_SUCCESS;
        } else {
            // 情况 A: 锁竞争 (tryLock 返回 false)
            // 另一个 Consumer 线程已持有此订单锁 → 退避重试
            return RECONSUME_LATER;
        }
    } catch (RedisConnectionException | RedisException e) {
        // □□□ 情况 B: Redis 宕机 (tryLock 抛异常) □□□
        // 这是与"锁竞争"物理上完全不同的失败模式
        // 必须走 DB 悲观锁降级路径，而不是 RECONSUME_LATER
        log.error("Redis down, falling back to DB pessimistic lock for order={}",
                  orderId, e);
        redisFailed = true;
    }
    
    // □□□ 降级执行流: 不使用 Redisson，直接用 MySQL 悲观锁 □□□
    if (redisFailed) {
        processTimeoutWithPessimisticLock(orderId);  // SELECT FOR UPDATE 独占
        return CONSUME_SUCCESS;
    }

  processTimeoutWithPessimisticLock 实现:
    @Transactional
    void processTimeoutWithPessimisticLock(Long orderId) {
        Order order = mapper.selectByIdForUpdate(orderId);  
        // SELECT * FROM seckill_order WHERE id = ? FOR UPDATE
        // FOR UPDATE：锁住这一行——其他事务的 SELECT FOR UPDATE 必须等待
        // 两个 Consumer 同时处理同一订单 → 串行化
    
        if (order.status == PENDING_PAY) {
            redisTemplate.opsForValue().increment(stockKey);  // 如果 Redis 恢复，回池
            order.setStatus(TIMEOUT_CANCELLED);
            mapper.updateById(order);
        } else if (order.status == PAID) {
            // 已支付 → 什么都不做 (幂等)
        }
        // MySQL COMMIT → 行锁释放
    }

  □□□ 降级决策树 (物理可执行版本) □□□
    tryLock(100ms)
      ├── true        → Redisson 锁持有 → processTimeout() → CONSUME_SUCCESS
      ├── false       → 锁竞争 (Redis OK) → RECONSUME_LATER
      └── Exception   → Redis 宕机 → processTimeoutWithPessimisticLock()
                        → SELECT FOR UPDATE → CONSUME_SUCCESS
                        → (不重试，因为重试没有 Redis 锁，DB 锁已经兜底完成)

  RocketMQ 重试策略（Consumer 侧，仅用于锁竞争）:
    第 1 次返回 RECONSUME_LATER → delayLevel=3  (10s)
    第 2 次返回 RECONSUME_LATER → delayLevel=4  (30s)
    第 3 次返回 RECONSUME_LATER → delayLevel=5  (1min)
    第 4 次返回 RECONSUME_LATER → 写入 DLQ → 人工介入
    // 注意: catch 到 RedisException 走的是 DB 降级路径，不进入重试

第四层: 状态机"已支付不可逆转"原则
  Consumer 处理逻辑中最核心的防御:
    order = mapper.selectById(orderId)  // 或 SELECT ... FOR UPDATE
    if (order.status == PENDING_PAY) {
        redis.incrBy(stockKey, 1)       // 回池
        order.setStatus(TIMEOUT_CANCELLED)
        mapper.updateById(order)
    } else if (order.status == PAID) {
        // 用户已支付 → 什么都不做，直接 ACK
        // 即使这是第 N 次重试消息，结果不变
        return ConsumeConcurrentlyStatus.CONSUME_SUCCESS
    } else if (order.status == TIMEOUT_CANCELLED) {
        // 已被另一条消息处理过 → 直接 ACK
        return ConsumeConcurrentlyStatus.CONSUME_SUCCESS
    }
    // 核心: 已支付的订单，无论任何重试风暴，永远不会被回滚 ✓
```

## 5.3 Redis Key Flood 与 Dict Rehash 抖动

### 攻击场景推演

恶意脚本绕过前端动态 URL 校验（或直接抓包拿到 Token），对秒杀接口发起海量请求，每个请求携带不同的 `random_userId`。系统为每个 userId 执行以下 Redis 操作：

```
1. ZADD rate:limit:random_uid_001 <timestamp> <timestamp>   → 创建新 ZSet
2. EXPIRE rate:limit:random_uid_001 2                         → 2 秒后过期
3. EXISTS {goods:1}:record:random_uid_001                     → 创建新 Key (购买记录检查)
```

每秒 5 万个不同 userId → 每秒 5 万个新 ZSet Key → Redis 内部发生什么？

### 物理级灾难链条

```
1. redisObject 内存包装开销:
   每个 Key 在 Redis 内部被包装为 redisObject 结构 (16 字节) + SDS 字符串 (~50 字节)
   每 Key ≈ 80 字节开销
   5 万 Key/s × 80 字节 = 4 MB/s → 240 MB/分钟 → 14 GB/小时
   Redis 4GB maxmemory → 10+ 分钟内 OOM → maxmemory-policy=noeviction → 拒绝所有写入
   → 合法用户的购买记录也写不进去 → 系统整体不可用 ❌

2. Dict 渐进式 Rehash 引发的延迟雪崩:
   Redis 全局 Key 字典 (Dict) 初始大小 4，负载因子 > 1 时触发扩容
   扩容过程是渐进式 (Progressive Rehash):
     - 分配新 Dict (容量 × 2)
     - 每次 CRUD 操作附带迁移 N 个 bucket 到新 Dict
     - 迁移期间 → 每个请求都要遍历两个 Dict → 对延迟敏感的场景是灾难
   5 万次 ZADD + 5 万次 EXISTS → 在 Rehash 期间 → 单次操作从 < 0.1ms 膨胀到 2~5ms
   P50 延迟: 0.1ms → P99 延迟: 8ms → 秒杀引擎吞吐从 10 万 QPS 跌到 1 万 QPS ❌
```

### 防御体系——三道物理屏障

```
第一道: IP 级令牌桶（在 userId 之前拦截）—— 含 OOM 防御
  RateLimitFilter 执行顺序:
    1. ipRateLimiter.acquire(remoteIp)   // ← 先在 IP 层面限流
       → 每个 IP 每秒最多 20 个请求（独立于 userId）
       → 恶意脚本即使用随机 userId，IP 不变 → 被统一拦截
    2. 仅通过 IP 限流后，才执行 userId 级别的限流和业务操作
    3. 这就避免了为海量随机 userId 创建 Redis Key

  □□□ 致命陷阱一：无界 Map<String, GuavaRateLimiter> → 100 万 IP → OOM □□□
    错误实现:
      ConcurrentHashMap<String, RateLimiter> ipLimiters = new ConcurrentHashMap<>();
      RateLimiter rl = ipLimiters.computeIfAbsent(ip, k -> RateLimiter.create(20));
      rl.tryAcquire();

    黑产攻击: 100 万个秒拨代理 IP → Map 中创建 100 万个 RateLimiter 对象
      RateLimiter 内部结构: 1 个 SmoothBursty 对象(~48B) + 1 个 mutex + storedPermits
      每个 ≈ 200 字节 → 100 万 × 200B = 200 MB → 直接 OOM
      → 第一道屏障成为最先倒下的多米诺骨牌 ❌

  □□□ 致命陷阱二：Caffeine get(key, mappingFunction) 永不返回 null □□□
    错误实现:
      Cache<String, RateLimiter> cache = Caffeine.newBuilder().maximumSize(10000).build();
      RateLimiter rl = cache.get(ip, k -> RateLimiter.create(20));

    Caffeine API 物理语义:
      get(key, mappingFunction) ≡ "如果不存在，执行 mappingFunction 创建，放入缓存，返回"
      它绝对不可能返回 null（除非 mappingFunction 返回 null，但 Caffeine 会抛 NPE）
    
    灾难推演:
      缓存已满 10000 → 新 IP 到达 → W-TinyLFU 驱逐旧 IP
      → 为新 IP 执行 RateLimiter.create(20) → 放入缓存 → 返回满额令牌桶
      → 100 万个肉鸡 IP 攻击 → 100 万个 RateLimiter 被创建/驱逐/再创建
      → 每个恶意 IP 都拿到 20 次新鲜请求额度 → 限流防线物理击穿 ❌
      → JVM CPU 被高频驱逐算法 + 对象创建打满 ❌
    
  □□□ 正确实现：getIfPresent + 极小硬黑名单 □□□
    // 第一步: 检查硬黑名单 (命中的 IP 曾被判定为恶意)
    Cache<String, Boolean> ipBlacklist = Caffeine.newBuilder()
        .maximumSize(1000)           // 黑名单极小, 只存确定的恶意 IP
        .expireAfterWrite(60, TimeUnit.SECONDS)
        .build();
    if (ipBlacklist.getIfPresent(ip) != null) {
        return 403;  // 直接拒绝
    }

    // 第二步: 用 getIfPresent (不自动创建) 检查 IP 是否在限流缓存中
    Cache<String, RateLimiter> ipLimiters = Caffeine.newBuilder()
        .maximumSize(10_000)
        .expireAfterAccess(30, TimeUnit.SECONDS)  // 30 秒无访问 → 驱逐
        .build();

    RateLimiter rl = ipLimiters.getIfPresent(ip);
    if (rl == null) {
        // IP 不在缓存中 → 不创建 RateLimiter → 放行到下一层
        // □□□ 核心防御: 未知 IP 绝不分配满额令牌桶 □□□
        // 1 万个已知 IP 已在缓存中 (高频者被限流)
        // 其余 IP 全部穿透到 → Bloom Filter (否决随机 userId) → userId ZSet
        return NEXT_FILTER;  // 放行
    }

    // 第三步: IP 在缓存中 → 已有 RateLimiter → 正常限流
    if (!rl.tryAcquire()) {
        // 超频 → 升级到黑名单
        ipBlacklist.put(ip, true);
        return 403;
    }
    return NEXT_FILTER;

    // 缓存预热: 外部线程每秒扫描 access log, 对高频 IP 执行:
    //   ipLimiters.asMap().putIfAbsent(highFreqIp, RateLimiter.create(20));
    //   (手动 putIfAbsent 不会触发 get() 的 mappingFunction)
    
  □□□ 三层纵深防御的物理实相 □□□
    第一道 (IP 令牌桶): 仅对已缓存的 1 万个 IP 限流, 其余全放
      面对百万秒拨 IP → 无可防御 → 但也不自毁 (不 OOM, 不 CPU 打满)
    第二道 (Bloom Filter): 否决随机/不存在的 userId → 真正的主力防御
    第三道 (userId ZSet):  对 Bloom Filter 假阳性穿透的 IP+userId 做最终限流

第二道: 用户 Bloom Filter 快速否决
  预热阶段: 将所有合法 userId 写入 Redis Bloom Filter (RedisBloom 模块 或 Guava BF)
  请求到达时:
    if (!bloomFilter.mightContain(userId)) {
        return FAIL("用户不存在")  // ← 不创建任何 Redis Key，直接拒绝
    }
    // 仅通过 Bloom Filter 的用户才进入后续 Redis 操作

  注意: Bloom Filter 有假阳性(误判存在)但无假阴性(不会误判不存在)
       → 脚本的随机 userId 几乎 100% 被 Bloom Filter 否决 → Redis Key 零创建

第三道: Redis Key 数量主动监控
  @Scheduled(fixedRate = 5000)  // 每 5 秒
  void monitorKeyCount() {
      Long rateLimitKeyCount = redisTemplate.execute(
          "EVAL \"return #redis.call('keys','rate:limit:*')\" 0"
      );
      // 或使用 SCAN + COUNT（生产环境）
      if (rateLimitKeyCount > expectedActiveUsers * 1.5) {
          log.warn("疑似 Key Flood 攻击, rate:limit 键数={}", rateLimitKeyCount)
          // 触发: 动态降低 IP 限流阈值、延长限流窗口、或拉入手动风控
      }
  }
```

# 6. 关键设计决策记录（ADR）

## ADR-001：为什么用 Maven 多模块而非微服务

- **决策**：Maven 多模块单体，单进程部署
- **理由**：秒杀核心链路延迟敏感（目标 < 1ms Redis + < 50ms 端到端），跨进程 RPC 的序列化开销不可接受。双机物理环境单机 16GB，微服务进程过多争抢 CPU。面试场景下，清晰的模块边界比微服务更能体现架构能力
- **代价**：无法独立扩缩容各模块。若未来需要独立扩容，可将 seckill-engine 拆为独立服务

## ADR-002：为什么 Filter Chain 在应用层而非 Nginx

- **决策**：限流/校验逻辑在 Spring Boot Filter 中实现
- **理由**：限流需要访问 Redis（滑动窗口 + QPS 计数器），动态 URL 校验需要 HMAC 签名计算，这些逻辑在 Java 层比 Nginx Lua 更可维护。双机环境中 Nginx 部署在 Node A 同机，不节省网络跳数
- **代价**：恶意流量会先穿透到 JVM 层才被拦截，极端 DDoS 下 Spring Boot 本身可能被打垮。生产环境应前置一层 Nginx 连接数限制 + 部分限流规则上移至 Nginx Lua

## ADR-003：为什么单实例 Redis 不使用库存分桶

- **决策**：`seckill_deduct.lua` 直接操作单 Key `{goods:{id}}:stock`，不做分桶拆分
- **理由**：
  ```
  分桶在 Redis Cluster 中的意义:
    将热点 Key 分配到不同节点 → 分摊集群中单节点的 CPU 和网络带宽

  分桶在单实例 Redis 中的实际效果:
    Redis 是单线程模型——10 个桶的 DECR 和 1 个 Key 的 DECR
    都在同一个 CPU 核心上串行执行 → 没有任何性能收益
    额外成本: ThreadLocalRandom 随机数开销 + 10 倍 Key 内存 +
             库存预热拆分逻辑 + 回池时需追踪桶编号 +
             捡漏需要随机路由

  保留 Hash Tag 但不分桶:
    Key = {goods:{id}}:stock  (单 Key, 仍使用 Hash Tag)
    → 同 goodsId 的 stock 和 record Key 在 Hash Tag 作用下同槽
    → Lua 原子操作不受影响
    → 未来迁移到 Cluster 时, 不同 goodsId 天然分散到不同节点
       (多商品隔离), 如需单商品热点拆分再加桶即可
  ```
- **代价**：若单商品 QPS 超单机 Redis 极限（约 10 万+），需引入分桶。当前双机环境单机 Redis 远未到此量级

## ADR-004：为什么 Hash Tag 用 `{goodsId}` 而非 `{global}`

- **决策**：Lua 脚本跨 Key 原子操作的 Hash Tag 边界设为 goodsId
- **理由**：`{global}` 会将所有商品的所有 Key 集中在同一 Redis 节点，放弃多商品隔离。`{goodsId}` 在同商品内保证跨 Key 原子性，异商品间天然分散到不同 Slot/Node → 手机流量不冲击牙刷流量
- **代价**：同商品所有 Key 不可跨节点水平扩展。若单商品 QPS 超单机 Redis 极限（约 10 万+），需引入客户端本地预减 + 异步对齐

## ADR-005：为什么用 RocketMQ 事务消息而非 Transactional Outbox

- **决策**：使用 RocketMQ 原生事务消息（`sendMessageInTransaction` + `TransactionListener`），**彻底废弃 `seckill_outbox` 表和 `OutboxScheduler` 轮询**
- **理由**：
  ```
  Transactional Outbox 在高并发秒杀场景下是性能自毁:
    开闸瞬间 1 万订单涌入:
      → 1 万行 INSERT seckill_order + 1 万行 INSERT seckill_outbox（同事务，写入量翻倍）
      → OutboxScheduler 每秒 SELECT * WHERE status=0 LIMIT 100
      → 高频并发 INSERT 与高频 SELECT 范围查询同时打在 InnoDB 上
      → 聚簇索引页分裂 + 行锁/间隙锁竞争 → MySQL IOPS 打满
      → 整个订单入库链路卡死 ❌

  Outbox 是 Kafka/RabbitMQ 用户迫不得已的妥协方案。
  RocketMQ 拥有业内最强原生事务消息——放着核武器不用，去手搓土铳，等于自证对技术栈核心能力无知。

  RocketMQ 事务消息正确模型:
    Phase 1: sendMessageInTransaction → 半消息持久化到 Broker（Consumer 不可见）
    Phase 2: executeLocalTransaction → INSERT order → COMMIT / ROLLBACK
    Phase 3: 若 JVM 宕机未返回 → Broker 主动回调 checkLocalTransaction → SELECT order → 补判 COMMIT / ROLLBACK

    轮询负担从 MySQL 转移给 RocketMQ Broker（C++ 实现，百万 QPS 级吞吐）
    → MySQL 只做 INSERT order，单表操作，零轮询
    → 写入量减半（无 outbox 表），IOPS 大幅下降 ✓
    → JVM 宕机兜底由 Broker 主动回查，非定时器被动等待 ✓
  ```
- **代价**：与 RocketMQ 强耦合（但我们的技术栈已经是 RocketMQ，这是利用优势，不是增加耦合）
- **面试口述要点**：
  > "这里我选择 RocketMQ 事务消息而非本地消息表。因为 Outbox 在高并发下会引入 MySQL 轮询开销——每秒 SELECT + 高频 INSERT 会在 InnoDB 上形成行锁竞争。RocketMQ 的事务消息通过 Half Message + checkLocalTransaction 回查机制，把一致性检查的负担从数据库转移到了 Broker，这对写密集型场景是本质性的性能提升。"

## ADR-006：为什么引入 Redisson 而非 Jedis 的 SETNX

- **决策**：订单状态流转的关键区使用 Redisson RLock（Watchdog 自动续期），而非 Jedis SETNX + 固定 TTL
- **理由**：
  ```
  Jedis SETNX + TTL 30s:
    → 线程获取锁，执行 SQL
    → JVM Full GC 停顿 40s
    → Redis 锁 TTL 到期自动释放
    → 另一个线程获取锁，同一订单被并发修改
    → 已支付订单被超时 Consumer 覆盖为 CANCELLED（生产灾难）

  Redisson RLock + Watchdog:
    → 线程获取锁
    → Watchdog 后台每 10s 续期一次（leaseTime / 3）
    → Full GC 40s → Watchdog 续期 4 次，锁始终有效
    → GC 恢复 → 线程正常执行 → unlock() → 锁释放
    → 无并发脏写 ✓
  ```
- **代价**：Redisson 依赖比 Jedis 重（Netty + 线程池），但对于有状态订单操作的模块，这个代价必须接受

## ADR-007：为什么用 Sentinel 熔断而非仅靠连接池超时

- **决策**：在连接池超时之上叠加 Sentinel 熔断器，形成"快速失败 + 自动恢复"双层防护
- **理由**：
  ```
  仅依赖连接池超时:
    Redis 连接池 max-wait=50ms → 每个请求等 50ms 后抛异常
    → 10 万并发 × 50ms = 5,000 线程秒/秒 → Undertow IO 线程迅速耗尽
    → 合法请求也被拖累 → 雪崩

  连接池超时 + Sentinel 熔断:
    慢调用比例 > 20% → OPEN (10 秒冷却)
    → OPEN 期间所有请求直接返回 fallback，不经过连接池
    → 10 万请求 × 0.001ms (fallback 是纯 CPU 操作) = 近乎零资源消耗
    → 10 秒后 Half-Open → 试探性放行 → 成功则 CLOSE，失败继续 OPEN
  ```
- **代价**：引入 Sentinel 依赖。但对于有外部中间件依赖的系统，熔断器是必备品

## ADR-008：为什么 Consumer 线程数刻意压低到 4

- **决策**：`OrderTimeoutConsumer` 最大线程数设为 4，而非默认的 20 或更高
- **理由**：
  ```
  高线程数的隐患:
    线程 = 资源消耗（1MB 栈空间 + CPU 上下文切换）
    20 线程同时处理超时订单 → 20 个 Redisson 锁竞争 + 20 个 MySQL SELECT FOR UPDATE
    → DB 连接池 (20) 瞬间耗尽 → 其他业务（如用户查订单）无法获取连接
    → 一个补偿任务拖垮整个系统

  4 线程的收益:
    峰值处理能力 = 4 / 100ms ≈ 40 条/秒
    100 条超时订单 ≈ 2.5 秒全部处理完毕
    超时订单晚 2.5 秒释放库存 vs 系统被拖垮 → 选择前者（业务无损）
    关键认知: 超时补偿是"后台保洁"，不是"前台交易"。后台任务永远不应争抢前台资源
  ```
- **代价**：大规模超时（如 10 万订单同时超时）时处理延迟增加。但此种场景在实际业务中几乎不存在（秒杀商品件数少），若出现，可动态扩容 Consumer 实例

## ADR-009：为什么防御顺序是 IP → Bloom → userId 而非 userId → IP

- **决策**：Gateway Filter Chain 中的限流顺序为 IP 令牌桶 → Bloom Filter → userId 滑动窗口，不可颠倒
- **理由**：
  ```
  错误顺序 (userId 优先):
    请求 → 按 userId 创建 Redis ZSet → 按 IP 限流
    → 恶意脚本 5 万随机 userId/s → 5 万 Redis Key/s 创建
    → Redis Dict Rehash + OOM → 系统瘫痪 ❌

  正确顺序 (IP 优先):
    请求 → IP 令牌桶 (Guava 本地内存，不碰 Redis)
      → 超频 → 直接拒绝（零 Redis 消耗）
      → 未超频 → Bloom Filter 否决随机 userId（零 Redis 消耗，误判时允许穿透）
      → 合法 → 创建 userId ZSet（此时已是可控量级）
    → Redis Key 创建量 = 合法用户量，而非攻击量 ✓
  ```
- **代价**：Bloom Filter 存在假阳性（~1% 误判存在），少数攻击 userId 可能穿透。但结合 IP 限流和第 3 步的 userId ZSet 异常 size 检测，形成纵深防御

## ADR-010：为什么 TCP 熔断必须从 OS 内核参数开始

- **决策**：调优 `somaxconn=65535` + `tcp_abort_on_overflow=1`，将 TCP 物理熔断下沉到操作系统层
- **理由**：
  ```
  错误的"TCP 熔断"认知:
    连接池超时 → Sentinel 熔断 → try-catch → 这些全部在 JVM 内部执行
    → 当 somaxconn=128(默认) 且 10 万并发到达时, OS 内核在第 129 个连接就开始静默丢包
    → 被丢的客户端在 SYN 重传超时 (60~120s) 中死等
    → 没有任何 Java 代码被执行 → 所有 JVM 层熔断机制形同虚设

  正确的物理熔断链:
    somaxconn=65535 → 10 万并发下 accept 队列不溢出
    tcp_abort_on_overflow=1 → 即使溢出, 内核直接回 RST (<1ms)
    → 客户端立刻感知 "Connection reset" → 可立即重试或降级
    → JVM 层 Sentinel/try-catch 才有机会真正工作

  认知边界: 连接池只管 JVM 内部连接, 不管 OS TCP 队列。TCP 队列满时, 连接根本没进 JVM
  ```
- **代价**：RST 返回比静默丢包更"暴力"，但对于秒杀场景，让客户端立刻知道失败是更优的选择

## ADR-011：为什么 IP 限流缓存必须用 Caffeine 而非 ConcurrentHashMap

- **决策**：`IpRateLimitFilter` 中的 IP→RateLimiter 映射使用 Caffeine `maximumSize(10000)` 有界缓存，而非无界 `ConcurrentHashMap`
- **理由**：
  ```
  无界 Map 的 OOM 灾难:
    ConcurrentHashMap<String, RateLimiter> 在 100 万秒拨 IP 攻击下:
    → 100 万个 RateLimiter 对象 (每个 ~200B) = 200 MB
    → 加上 Map Entry + String Key 开销 ≈ 500 MB+
    → 16 GB 堆中形成巨大的老年代碎片
    → Full GC 频繁触发 → 秒杀 P99 延迟飙升 → 更严重时直接 OOM

  Caffeine 有界缓存的防御:
    maximumSize(10000) → 物理硬上限, JVM 内存使用恒定
    expireAfterWrite(30s) → 攻击 IP 停止后自动释放空间
    W-TinyLFU 驱逐策略 → 高频攻击 IP 留在缓存中持续被限流
                         低频 IP 被驱逐 → 放行至 Bloom Filter → 随机 userId 被否决
    关键认知: 宁可让溢出的恶意 IP 穿透到下一层, 也绝不让 JVM 内存防线崩溃
  ```
- **代价**：缓存满时 ~1% 的恶意 IP 可能穿透 IP 限流层。由 Bloom Filter (第二道) + userId ZSet (第三道) 兜底

## ADR-012：为什么用 getIfPresent 而非 get(mappingFunction)

- **决策**：IP 限流缓存使用 `Caffeine.getIfPresent(ip)` 而非 `get(ip, k -> RateLimiter.create(20))`
- **理由**：
  ```
  get(key, mappingFunction) 的物理语义:
    key 不存在 → 执行 mappingFunction → 创建新对象 → 放入缓存 → 返回
    这个 API 永远不会返回 null（mappingFunction 返回 null 则抛 NPE）
    
  对 IP 限流的致命后果:
    缓存满 10000 → 新 IP 到达 → 驱逐旧 IP → 为新 IP 执行 RateLimiter.create(20)
    → 100 万秒拨 IP 攻击 → 100 万个满额令牌桶被创建 → 限流防线物理击穿
    → JVM CPU 被高频驱逐算法打满

  getIfPresent(ip) 的物理语义:
    key 存在 → 返回 value; key 不存在 → 返回 null
    绝不自动创建 → 未知 IP 无令牌桶 → 放行至下一层 (Bloom Filter)

  防御分工:
    IP 令牌桶: 仅限流已缓存的已知高频 IP (<= 10000 个)
    Bloom Filter: 否决随机 userId (百万级 IP 的主力防御)
    userId ZSet: 最终兜底
  ```
- **代价**：需要独立的缓存预热机制（从 access log 识别高频 IP，手动 `putIfAbsent`）。秒拨 IP 的单个低频请求直接放行

## ADR-013：为什么 tryLock 失败需要区分 false vs Exception

- **决策**：`lock.tryLock(100ms)` 的 false 和 Exception 是两种物理上完全不同的失败模式，必须显式分流
- **理由**：
  ```
  错误代码（PPT 架构）:
    if (lock.tryLock(100ms)) { ... } else { return RECONSUME_LATER; }
    // "即便 Redisson 不可用，MySQL 行级锁仍然兜底" ← 这句注释是谎言
    // 因为 tryLock false/Exception 后方法已经 return，永远走不到 DB 锁代码

  物理上完全不同的两种失败:
    tryLock 返回 false:
      → Redis 正常工作，但锁被另一个 Consumer 线程持有
      → 退避重试 (RECONSUME_LATER) 是正确的

    tryLock 抛出 RedisConnectionException:
      → Redis 宕机，不是锁竞争问题
      → RECONSUME_LATER 会导致消息无限重试堆积，永不成功
      → 正确的处理: catch(Exception) → 跳过 Redisson → SELECT FOR UPDATE 兜底
      → 不重试（DB 锁已经保证了原子性）

  降级决策树:
    tryLock(100ms)
      ├── true        → Redisson 持有 → 正常处理 → CONSUME_SUCCESS
      ├── false       → 锁竞争 → RECONSUME_LATER (延迟重试)
      └── Exception   → Redis 宕机 → SELECT FOR UPDATE → CONSUME_SUCCESS
  ```
- **代价**：DB 降级路径依赖 MySQL 行级锁性能。单行 FOR UPDATE 性能可接受（毫秒级），但不能替代 Redisson 的高频使用

## ADR-014：为什么订单 ID 必须用 Snowflake 而非 AUTO_INCREMENT

- **决策**：`seckill_order.id` 由应用层 Snowflake 提前生成，废除 MySQL `AUTO_INCREMENT`
- **理由**：
  ```
  RocketMQ 事务消息的时空约束:
    Phase 1: 发送 Half Message → 消息体中必须包含订单 ID（Consumer 需要它定位订单）
    Phase 2: 执行本地事务 → INSERT seckill_order(id, ...)
    Phase 3: COMMIT / ROLLBACK

    若用 AUTO_INCREMENT:
      → 发半消息时还不知道 ID → 消息体无法携带订单 ID
      → Consumer 收到延时消息后不知道查哪条订单 → 整个超时回收链路断裂 ❌

    若用 Snowflake:
      → Phase 0 预生成 orderId = snowflake.nextId()
      → Phase 1 发送 Half Message (含 orderId)
      → Phase 2 INSERT INTO seckill_order (id, ...) VALUES (orderId, ...)
      → Consumer 直接用消息体中的 orderId 查订单 ✓

  面试官追问 "Snowflake 在双机环境下会冲突吗":
    10bit 机器ID → 最多 1024 个节点
    Node A (seckill-server):   workerId=1, datacenterId=1
    Node B (MySQL / JMeter):   不产生订单 ID
    → 单机生成, 零碰撞 ✓
  ```
- **代价**：Snowflake ID 按时间趋势递增但非严格连续，InnoDB 聚簇索引页分裂略多于自增。性能差异 < 5%，可接受

## ADR-015：为什么移除 uk_user_goods 唯一索引

- **决策**：`seckill_order` 不设 `UNIQUE KEY (user_id, goods_id)`，仅设普通索引 `INDEX idx_user_goods`
- **理由**：
  ```
  业务需求: 超时未支付的订单自动取消 (status=3)，库存回池后用户可再次抢购（捡漏）

  若有唯一索引:
    用户 A 抢购 → 超时取消 (status=3) → 再次抢购成功
    → Redis 层扣减成功 → 异步 INSERT 新订单
    → MySQL: DuplicateKeyException → 合法复购被物理拒绝 ❌

  若仅普通索引:
    用户 A 超时订单 (id=100, status=3) 存在
    用户 A 再次抢购 → Redis Lua DELETE 旧记录 + SET 新记录 (原子)
    → 异步 INSERT 新订单 (id=200, status=0) → 两行共存 ✓

  防重复怎么办:
    完全由 Redis Lua 脚本的 EXISTS {goodsId}:record:{userId} 原子校验承担
    → 该检查在库存扣减之前执行, 同一秒内不可能通过两次
    → MySQL 索引 idx_user_goods 仅用于"查询用户是否已下单"
  ```
- **代价**：MySQL 层面失去物理防重兜底。若 Redis 被重启且 RDB 丢数据，极端情况下可能出现重复订单。生产环境可通过 Redis AOF 持久化 + 定时对账脚本兜底

## ADR-016：为什么金额字段用 INT 而非 DECIMAL

- **决策**：所有价格字段（`original_price`、`seckill_price`）使用 `INT UNSIGNED`，单位为分
- **理由**：
  ```
  DECIMAL 的单位语义矛盾:
    注释写"原价（分）" → 单位是分 → 物理值是整数
    类型写 DECIMAL(10,2) → 暗示有小数位 → 单位应是元
    → 开发时极易搞混 ("99.99 元" vs "9999 分")

  工程规范（微信支付/支付宝一致）:
    金额用 INT BIGINT → 单位到分 (或厘)
    → 无浮点精度问题 (100 元 = 10000 分, 精确)
    → JSON 序列化无精度损失 (Java long ↔ JSON Number ↔ JS)
    → CPU 做整数运算比 DECIMAL 快约 3 倍
    → Redis Lua 中 tonumber() 解析整数无歧义
    → 跨语言 (Java/Lua/Go/JS) 全链路一致

  前端展示: price / 100.0 → "99.99 元" (仅展示层转换)
  ```
- **代价**：前后端需约定单位为分。`application.yml` 中加 `price-unit: cent` 配置项并写入 API 文档

# 7. 数据库表结构设计

## 7.1 设计原则

```
1. MySQL 仅做最终持久化，不参与秒杀核心路径
   → 库存扣减在 Redis 完成，MySQL stock 字段仅用于预热加载和最终对账

2. 秒杀路径零写 MySQL
   → 下单是异步的（seckill-engine 返回 SUCCESS 后，MQ/线程池异步写 MySQL）
   → 核心路径的延迟仅依赖 Redis 网络往返 + Lua 执行

3. 索引服务于"查询"，防重复下沉到 Redis
   → 最频繁的查询：用户是否已下单（不依赖唯一约束，允许超时后复购）
   → 防重复完全由 Redis Lua 原子 EXIST 校验承担
   → 不在 MySQL 层面做库存递减（UPDATE stock = stock - 1 WHERE stock > 0）
   → 不在 MySQL 层面做防重复唯一约束（超时订单的回购需要新记录）
```

|------|------|------|------|
| `id` | BIGINT UNSIGNED | PK | 订单 ID（Snowflake 预生成，见 ADR-014） |
| `user_id` | BIGINT UNSIGNED | NOT NULL | 用户 ID |
| `goods_id` | BIGINT UNSIGNED | NOT NULL, FK → seckill_product.id | 商品 ID |
| `activity_id` | BIGINT UNSIGNED | NOT NULL, FK → seckill_activity.id | 活动 ID |
| `seckill_price` | INT UNSIGNED | NOT NULL | 秒杀成交价（单位:分，下单快照） |
| `status` | TINYINT | NOT NULL DEFAULT 0 | 0-待支付 1-已支付 2-已取消 3-超时取消 |
| `create_time` | DATETIME | NOT NULL DEFAULT CURRENT_TIMESTAMP | 下单时间 = 支付倒计时起点 |
| `pay_time` | DATETIME | DEFAULT NULL | 支付完成时间 |
| `cancel_time` | DATETIME | DEFAULT NULL | 取消时间 |
| `update_time` | DATETIME | ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**索引**：
```sql
INDEX idx_user_goods (user_id, goods_id)        -- 用户是否已购买（非唯一，允许超时后复购）
INDEX idx_status_ctime (status, create_time)     -- 超时消费者查询待支付订单
INDEX idx_activity (activity_id)                 -- 按活动统计
```

**说明**：
- □□□ 不设 UNIQUE (user_id, goods_id) —— 超时回池后用户可再次抢购（捡漏），MySQL 允许同一用户对同一商品存在多条历史订单（status 不同）。防重复完全由 Redis Lua 脚本的 `EXISTS {goodsId}:record:{userId}` 原子校验保证。此乃面试中必须主动解释的架构决策。
- `seckill_price` 冗余存储下单瞬间的价格快照（单位:分），避免后续商品改价影响历史订单。

## 7.4 核心查询场景与索引覆盖分析

| 场景 | SQL | 命中索引 | 说明 |
|------|-----|---------|------|

| 超时订单扫描 | `WHERE status=0 AND create_time < NOW() - INTERVAL 5 MINUTE` | `idx_status_ctime` | checkLocalTransaction 或消费失败后的兜底扫描 |
| RocketMQ check 回查 | `WHERE id = ?` | PRIMARY KEY (Snowflake ID) | checkLocalTransaction 回查——点查询，毫秒级 |
| 用户成功购买确认 | `WHERE user_id=? AND goods_id=? AND status=1` | `idx_user_goods` | 幂等性校验 |
| 活动销售统计 | `WHERE activity_id=? AND status IN (0,1)` | `idx_activity` | 运营后台 |

## 7.5 Redis Key 命名规范与防弹约束

> 与 MySQL 表结构配套，统一 Key 空间设计。每个 Key 附带高并发下的防护约束。

### 7.5.1 Key 目录

```
# ── 秒杀核心 Key（使用 Hash Tag，单机与 Cluster 一致）──

# 商品库存（预热时写入，单 Key 无分桶）
{goods:{id}}:stock                → INT
示例: {goods:1}:stock             → 100
约束: 预热后只减不增（除回池 INCRBY），不设 TTL

# 商品售罄标志（O(1) 列表页查询，异步写入）
{goods:{id}}:sold_out              → "1"     (EX 60)
示例: {goods:1}:sold_out           → "1"
约束: seckill_deduct.lua 检测 DECR 后 stock=0 时异步 SET
     列表页读此标志 + Caffeine 缓存(2s)

# 用户购买记录（秒杀成功时写入，Lua 原子操作）
{goods:{id}}:record:{userId}      → "1"     (EX 3600)
示例: {goods:1}:record:888        → "1"
约束: 必须设置 TTL（活动结束后自动清理），值固定为 "1"（最小化内存）

# 限购一件全局占位（跨商品场景）
{global}:record:{userId}          → "{goodsId}"  (EX 3600)
示例: {global}:record:888         → "1"
约束: 值存 goodsId 用于补偿阶段定位回滚目标

# ── 网关层 Key（不使用 Hash Tag，无跨 Key 需求）──

# 滑动窗口限流计数（按用户，ZSet 实现）
rate:limit:{userId}               → Sorted Set (ms 时间戳为 member 和 score)
示例: rate:limit:888              → ZSet [1715472000000, 1715472000100, ...]
约束: 见 6.5.2 大 Key 防护

# 当前 QPS 水位（全局计数器）
rate:watermark:qps                → INT     (EX 1, INCR)
示例: rate:watermark:qps          → 8734
约束: 单 Key，Redis 单线程 INCR 无竞争，无需特殊处理

# 动态抢购 Token（一次性，验证后 DEL）
token:{token}                     → "{goodsId}:{userId}"  (EX 60)
示例: token:a1b2c3d4              → "1:888"
约束: 一次性使用，验证通过立即 DEL，防止重放
```

### 7.5.2 大 Key / 热 Key 防护约束

**问题场景：ZSet 限流 Key 的膨胀风险**

`rate:limit:{userId}` 使用 ZSet 存储毫秒级时间戳。恶意脚本若以每秒 100 次频率攻击，单个 ZSet 在 1 秒窗口内即膨胀至 100 个 member。如果攻击持续且清理逻辑失效，ZSet 会无限膨胀为 **大 Key**，在 Redis 单线程模型下引发数毫秒甚至数十毫秒的阻塞，拖垮整个秒杀集群。

**强制防护措施**（在 Lua 限流脚本中硬编码，不可绕过）：

```
Lua 限流脚本内置三合一防护:
  1. ZREMRANGEBYSCORE key 0 (now - window_ms)  -- 先清理过期 member（每次请求都执行）
  2. ZCARD key                                   -- 统计窗口内请求数
  3. 若 member 数 > 窗口阈值 × 2（异常判定）:
       → 直接拒绝（不追加新 member）
       → 记录告警日志（此 userId 疑似脚本攻击）
  4. ZADD key now now                            -- 追加本次请求
  5. EXPIRE key window_seconds                   -- 续期 TTL（无请求时自动过期删除）

TTL 兜底:
  - 窗口 1 秒 → EXPIRE 2 秒（2 倍窗口，留足余量）
  - 即使 Lua 第 5 步执行失败，Key 最晚 2 秒后自动删除
  - 杜绝僵尸 Key 堆积
```

**验证标准**：压测时 `redis-cli --bigkeys` 和 `MEMORY USAGE rate:limit:*` 抽样，确保无单个 ZSet member 数超过 200、无单个 Key 内存超过 10KB。

### 7.5.3 内存淘汰策略——必须 `noeviction`

**致命场景**：购买记录 `{goods:1}:record:888` 带有 3600 秒 TTL。秒杀洪峰期 Redis 内存若被打满，默认的 LRU/LFU 淘汰策略可能**提前删除**此 Key。后果：

```
用户 A 秒杀成功 → {goods:1}:record:888 = "1" (EX 3600)
    → 内存压力 → Redis 淘汰此 Key
    → 用户 A 再次请求 → Lua 检查购买记录 → 不存在
    → 再次扣减库存 → 用户 A 成功购买第 2 件
    → 限购一件被击穿 ❌
```

**强制约束**：

```yaml
# redis.conf 或运行时 CONFIG SET
maxmemory-policy noeviction
# 含义: 内存写满后拒绝所有写入请求并返回 OOM 错误
# 宁可拒绝新请求，绝不淘汰已有购买记录
```

**配套措施——内存余量计算**：

```
单商品预估内存 = 购买记录数 × 单 Key 大小
              = 最终成功用户数 × (Key 名长度 + Value 1 字节 + Redis 内部开销 ~80 字节)
              = 100 × 120 ≈ 12 KB（可忽略）

实际风险点不是购买记录，而是限流 ZSet。
限流 ZSet 峰值 = 并发用户数 × 窗口内最大请求数 × (member 8 字节 + score 8 字节 + 开销)
                = 10 万 × 10 × 50 ≈ 50 MB

预留策略: maxmemory ≥ 预估峰值 × 2（留 1 倍安全余量）
          Node A 16GB → 分配 4GB 给 Redis → maxmemory 4GB → 余量充足
```

## 7.6 Redis 部署策略：单机运行，集群就绪

### 现状

测试与压测环境的物理拓扑是 **单台 Node A 运行单机 Redis-SERVER 进程**（非 Cluster 模式）。

### 为什么保留 Hash Tag 设计

```
表面上: 单机 Redis 不存在跨 Slot 问题，Hash Tag 是多余的
实际上: Hash Tag 是一种"架构约束语法"，它强制 Lua 脚本在 Key 维度
        上诚实，杜绝了未来迁移到 Cluster 时的 CROSSSLOT 灾难

迁移路径:
  测试环境:  redis-server (standalone, 单机)
              └── Hash Tag 语法兼容，无实际分片行为
              └── 所有 Key 天然在同一实例，Lua 正常执行

  生产环境:  Redis Cluster (3 主 3 从)
              └── {goods:1}:* 落到 Slot X → Node A
              └── {goods:2}:* 落到 Slot Y → Node B
              └── CROSSSLOT 物理隔离天然生效
              └── 零代码修改，仅改 Redis 连接地址
```

### 面试中的价值表达

> "我的测试环境是单机 Redis，但我全量使用了 Hash Tag 语法。这是刻意为之——它让 Lua 脚本在 Key 设计层面就遵守了 Cluster 的跨槽约束。未来从单机迁移到集群，代码一行不用改，只需换连接串。这个习惯来自我第一次把单机项目迁移上生产时，被 CROSSSLOT 错误打垮的教训。"

## 7.7 配置分层：业务数据 vs 运行时参数

| 类型 | 存储位置 | 示例 | 变更成本 |
|------|---------|------|---------|
| 业务数据 | MySQL | 活动时间、商品名称、价格、库存 | 需管理后台操作 |
| 物理运行时参数 | `application.yml` / Nacos | `rate-limit.threshold`、`rate-limit.user-qps` | 改配置重启（或 Nacos 热更新） |
| 临时令牌 | Redis（TTL 自动过期） | 动态 URL Token、验证码 Ticket | 零运维成本 |

**关键边界**：库存直接写入 Redis 单 Key，无分桶。单实例 Redis 下分桶无性能收益（单线程串行执行），只增加代码复杂度。
