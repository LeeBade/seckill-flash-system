# 需求设计（业务场景描述）

秒杀是**一场用极小成本换取极大流量的营销活动。** 这决定它有两条不可逾越的红线：
1. **绝对不能超卖**
2. **绝对不能影响其他业务**


## 场景 1：活动预热与倒计时阶段

- **服务端倒计时统一**：用户看到的开抢倒计时必须以服务端时间为准，客户端本地时间不可信。倒计时归零前，任何抢购请求均为无效。
- **购买入口动态化**：抢购的真实 URL 在倒计时结束瞬间才由服务端生成并下发。未到时间点，任何人无法提前构造或猜解购买链接，从源头阻断脚本批量预请求。
- **多商品独立运营**：同一场活动可同时售卖多个商品（如 100 台手机、200 个台灯、400 个牙刷），每种商品有独立库存。热门商品的抢购压力不得波及冷门商品用户的正常购买体验。
- **全场限购一件**：每个用户在整个活动期间只能成功购买一件商品。若用户同时发起多商品抢购，仅保留最先命中库存的那一单，其余全部拦截。抢购失败后，用户可继续尝试购买——但一旦成功，立即锁定，不可再抢。

---

## 场景 2：开闸瞬间——海量并发与库存争夺

- **10 万人抢 100 件，毫秒级判决**：开闸瞬间约 10 万并发请求涌入，争抢仅 100 件真实库存。系统须在毫秒内完成全部判决——抢到的进入下单流程，抢不到的（约 99,900 人）立刻收到明确的"已售罄"结果，禁止出现转圈等待或排队阻塞。
- **热门商品的负荷分散**：对于预期极度抢手的单品，将全部库存预拆为多个等额份额并分散管理。每位用户被随机分配到一个份额组内竞争。原本 10 万人争一个入口，变为 10 个 1 万人的小组各自竞争，从而避免单一热点被打垮。
- **库存扣减的绝对正确性**：任何场景下、任意并发程度下，"已售数量 ≤ 实际库存"这一约束必须数学级严格成立。不允许出现两人同时抢到最后一件的情况。

---

## 场景 3：下单后的支付时效与库存回收

- **限时支付锁定**：抢到购买资格后，该件商品被暂时锁定。用户须在规定时间（如 5 分钟）内完成支付。超时未支付，订单自动取消，锁定即刻解除。
- **静默回池与"捡漏"**：超时释放的库存不主动通知任何人，而是直接、静默地退回可售库存池。此时仍在反复尝试下单的用户，有几率直接命中这笔回流库存，实现自然的"捡漏"交易。
- **超时自动释放**：超时订单的取消与库存回池必须自动完成，无需人工或定时脚本介入。

---

## 场景 4：恶意流量与反欺诈

- **高频访问自动拦截**：对同一用户或同一来源的异常高频请求，在入口处即予以熔断，不得穿透至后端交易系统。
- **流量水位自适应验证**：当系统整体负载触及警戒线时，自动对后续请求追加人机验证（算术题 / 滑块验证码）。利用真人完成验证的 1~3 秒自然耗时，将瞬间峰值的请求在时间轴上均匀摊开，实现平滑削峰。负载回落至安全水位后，自动取消验证，恢复直通。
- **验证交互的无感融合**：用户在抢购按钮上仅需点击一次。若触发验证，验证组件自动弹出；验证通过后系统自动续接原抢购流程。全程无需用户刷新页面或重复点击，在拦截脚本的同时保障真人体验的流畅性。

# 场景面试重要性与技术实现分析

> 以下对每个业务场景评估其在 Java 工程师面试中的**重要性**（高/中/低），并给出为实现该场景所需的**核心技术栈**与**数据流转链路**。


## 场景 1：活动预热与倒计时阶段

### 1.1 服务端倒计时统一
- **面试重要性**：★★☆☆☆（低）
- **理由**：技术实现简单，本质上是一次 HTTP 时间查询。面试官通常不会深挖此点，但可作为系统完整性的注脚。
- **核心技术**：Spring Boot REST 接口、NTP 时间同步
- **数据流转**：
  ```
  客户端 --GET /server-time--> 网关 --> Spring Boot Controller --> 返回服务端 System.currentTimeMillis()
  ```
- **面试展开点**：可引申为客户端与服务端时钟不一致时如何做数据一致性保证（如分布式 ID 生成、TOTP 等）。

### 1.2 购买入口动态化
- **面试重要性**：★★★☆☆（中）
- **理由**：体现基础的 Web 安全思维，是防脚本、防重放攻击的入门案例。
- **核心技术**：HMAC 签名 / MD5 Salt、请求签名验证拦截器（Spring Interceptor / Filter）
- **数据流转**：
  ```
  1. 倒计时归零 → 前端 fetch("/api/generate-token?goodsId=1")
  2. Controller 根据 goodsId + userId + 服务端密钥 生成签名 Token
  3. 前端拼接真实 URL: /api/seckill?goodsId=1&token=xxx&userId=yyy
  4. 网关 Filter 拦截 → 校验 Token 合法性 → 不合法直接 403
  5. 合法 → 放行至秒杀核心
  ```

### 1.3 多商品独立运营
- **面试重要性**：★★★★☆（高）
- **理由**：直接触及分布式系统的"资源隔离"核心命题。面试官会深挖"热点如何影响非热点""如何做物理或逻辑隔离"。
- **核心技术**：Redis 多实例 / 分片、线程池隔离（Hystrix/Sentinel）、Tomcat 连接池分区
- **数据流转**：
  ```
  请求 --> 网关路由 --> 按 goodsId 分发至独立的业务线程池 / Redis 分片
                      
  商品A（热门/高并发）----> Redis 分片 A（独立连接池）----> 独立扣减逻辑
  商品B（冷门/低并发）----> Redis 分片 B（独立连接池）----> 独立扣减逻辑
  
  // 关键：A 的线程池满不影响 B 的可用连接
  ```

### 1.4 全场限购一件（跨商品并发对抗）
- **面试重要性**：★★★★★（非常高）
- **理由**：跨商品原子性校验是秒杀系统中并发控制难度最高的场景之一。涉及"检查-扣减"的原子化、分布式并发竞态、以及失败后的状态回滚，是面试官筛选候选人的分水岭问题。
- **核心技术**：Redis SETNX（分布式锁替代方案）、Lua 脚本原子化、两阶段提交（Saga 补偿）、Hash Tag 强制同槽
- **数据流转**：
  ```
  关键前置决策：跨商品全局记录与单商品库存不可能天然落在同一 Redis Cluster Slot。
  方案选择——集中式全局记录节点：
    使用固定 Hash Tag {global} 强制所有用户购买记录落在同一节点
    商品库存 Key 使用各商品独立 Hash Tag {goodsId} 实现多商品分片隔离
    两阶段操作 + 补偿回滚：

  用户并发发起 3 个请求（商品A、B、C）
       │
       ▼
  阶段一（集中节点 {global}，Lua 原子）:
    SETNX {global}:record:{userId} EX 3600  ← 原子抢占"全局限购位"
    ├── 返回 0 → 已被抢占 → 返回 "您已购买过"（拒绝）
    └── 返回 1 → 抢占成功 → 进入阶段二

  阶段二（商品分片节点 {goodsId}，Lua 原子）:
    DECR {goodsId}:stock:bucket{N}
    ├── >=0 → 扣减成功 → 返回 "抢购成功"
    └── <0  → 扣减失败 → 触发补偿:
              INCR {goodsId}:stock:bucket{N}   (库存回滚)
              DEL {global}:record:{userId}     (释放限购位)
              返回 "库存不足"

  并发 3 请求中，阶段一仅 1 个能 SETNX 成功，其余 2 个直接拒绝
  // 面试要点：坦诚承认分布式场景下无法做到完美单阶段原子性，
  // 但通过 Saga 补偿 + 严格顺序（先占位→再扣减→失败则释放）保证最终一致性
  ```

## 场景 2：开闸瞬间——海量并发与库存争夺

### 2.1 毫秒级判决与快速失败
- **面试重要性**：★★★★★（非常高）
- **理由**：这是秒杀系统的灵魂——"3 秒内出结果，绝不阻塞"。面试官会追问：为什么不排队？为什么不用消息队列削峰？快速失败的边界在哪里？
- **核心技术**：Redis 原子操作（O(1) 复杂度）、Tomcat/Undertow 异步非阻塞配置、连接池最大连接数调优、Netty 反应式编程
- **数据流转**：
  ```
  10万并发请求
     │
     ▼
  Nginx / Gateway (最大连接数限制，超额直接 RST)
     │
     ▼
  Spring Boot (Undertow IO 线程 → 业务线程池，线程数 = CPU 核心数)
     │
     ▼
  Redis Lua 脚本 ({goodsId} Hash Tag 同槽，单线程串行执行，微秒级)
     │
     ├── 库存命中 (100个) → 异步写 MQ 生成订单 (极速响应用户 "抢到了")
     └── 库存为0 (99900个) → 直接返回 "已售罄" (极速响应)
  
  // 关键指标：每个请求在 Redis 层面的处理时间 < 1ms
  // 99900 个失败请求不会进入任何等待队列
  ```

### 2.2 热门商品库存分片
- **面试重要性**：★★★★★（非常高）
- **理由**：从单点瓶颈到分布式分治——这是高级工程师与初中级工程师的分界线。面试官会考察路由策略的选择依据、分片后的一致性问题、以及路由与业务的相容性。
- **核心技术**：随机路由策略（非一致性 Hash）、Redis Cluster / 分片分桶、Hash Tag `{goodsId}` 强制同槽
- **路由策略选择——为什么不用一致性 Hash：**
  ```
  一致性 Hash(userId) → 用户被永久钉死在固定 Bucket
    → Bucket-0 售罄后，该用户永远无法触碰 Bucket-1~9 的回流库存
    → 直接杀死 3.2 节的"捡漏"机制 ❌

  随机路由 → 每次请求随机选择 Bucket
    → 统计上负载均匀（10 桶 × 随机 ≈ 各 10%）
    → 用户重试时可能命中任意桶 → 捡漏机制存活 ✓
    → 省去 Hash 计算开销 → 更快 ✓
  ```
- **数据流转**：
  ```
  库存 100 件 → 预热拆分为 10 个桶，每桶 10 件:
    Bucket-0: key = "{goods:1}:stock:bucket0", value = 10 → Slot X → Node A
    Bucket-1: key = "{goods:1}:stock:bucket1", value = 10 → Slot X → Node A
    ...
    Bucket-9: key = "{goods:1}:stock:bucket9", value = 10 → Slot X → Node A

    // 注意：所有 Bucket 使用相同 Hash Tag {goods:1}，
    // 因此强制落在同一 Slot → 同一节点 → 允许跨 Key Lua 原子操作（见 2.3）
    // 分桶的意义不是跨节点分布，而是拆分单 Key 热点（避免 Redis 单 Key 过载）
  
  请求到达网关:
    1. 随机数生成 BucketIndex (0~9)
    2. 拼接 Key: "{goods:1}:stock:bucket{BucketIndex}"
    3. 执行 Lua 原子扣减（见 2.3）
  
  10 万 QPS → 10 个分桶 Key（同一节点）→ 提升 Redis CPU 多线程利用率
  
  // 面试要点：Redis Cluster 的分片是按 Hash Slot 的，不是按 Key！
  // 同一 Slot 内的多个 Key 仍可被 Lua 原子操作。
  // 按 goodsId 分片实现多商品隔离（不同 goods → 不同 Slot/Node），
  // 按 bucket 分桶实现单商品内热点分摊（同 Slot 不同 Key）。
  ```

### 2.3 库存扣减的绝对正确性（防超卖）—— 含 CROSSSLOT 物理级修复
- **面试重要性**：★★★★★（非常高）
- **理由**：这是秒杀系统的底线命题。直接关联到"你会用什么机制保证不超卖？""Lua 脚本为什么是原子的？""Redis Cluster 下 Lua 跨 Key 有什么坑？"等连环追问。
- **核心技术**：Redis Lua 脚本（EVALSHA）、Hash Tag `{goodsId}` 强制同槽、Redis 单线程原子模型
- **致命陷阱——CROSSSLOT 物理报错**：
  ```
  错误写法（原方案——直接上线即崩溃）：
    KEYS[1] = "seckill:stock:goods1:bucket0"     → CRC16 → Slot 1234
    KEYS[2] = "seckill:record:user888"           → CRC16 → Slot 5678
    → Redis Cluster 返回: "CROSSSLOT Keys in request don't hash to the same slot"
    → Lua 脚本根本无法执行，全部请求物理拦截 ❌

  面试官追问："你怎么解决？"
  ```
- **修复方案——Hash Tag 强制同槽**：
  ```
  Redis Hash Tag 机制: Key 中第一个 { 和第一个 } 之间的部分用于计算 Slot
  其余部分不参与 Hash 计算。

  修正后的 Key 设计:
    KEYS[1] = "{goods:1}:stock:bucket3"    → CRC16("goods:1") → Slot X
    KEYS[2] = "{goods:1}:record:user888"   → CRC16("goods:1") → Slot X (相同!)
    → 两 Key 强制落在同一 Slot → Lua 原子执行 ✓
  ```
- **完整修正版数据流转**：
  ```
  Lua 脚本（同一 {goodsId} Slot 内原子执行）:

  local stock = redis.call('GET', KEYS[1])           -- {goods:1}:stock:bucket3
  local bought = redis.call('EXISTS', KEYS[2])        -- {goods:1}:record:user888
  if bought == 1 then
      return -1  -- 已购买过该商品
  end
  if stock and tonumber(stock) > 0 then
      redis.call('DECR', KEYS[1])                     -- 扣库存
      redis.call('SETEX', KEYS[2], 3600, '1')         -- 写购买记录
      return 1  -- 抢购成功
  else
      return 0  -- 库存不足
  end

  竞态场景验证:
    请求A: GET stock=1 → DECR → stock=0 → SETEX record ✓
    请求B: GET stock=0 → return 0 ✗  (Lua 单线程串行，B 在 A 之后)
  
  // Redis 单线程 + Hash Tag 同槽 + Lua 原子 = 数学级防超卖
  ```

- **架构权衡坦诚说明**：
  ```
  利：同一 goodsId 的所有 Key（库存桶 + 购买记录）原子操作 ✓，多商品隔离 ✓
  弊：同一 goods 的所有流量集中在一台 Redis 节点（单商品热点瓶颈）
  补救：
    1. 同节点内 10 个分桶 → 拆分 Key 级热点，Redis 单线程轮流处理各桶
    2. 若单商品 QPS 超出单机 Redis 极限（约 10万+），引入客户端本地预减 + 异步对齐
    3. 多商品场景下，不同 goodsId 天然分布在不同节点，实现商品级资源隔离
  
  面试官心理：你主动指出了方案的物理边界和改进路径 → 加分信号
  ```

### 2.4 Redis Cluster 物理级设计原则（兼答 CROSSSLOT 与路由拓扑冲突）

> 本节为 2.2~2.3 及 3.2 暴露出的两个架构级冲突提供统一解释和面试应答策略。

**原则一：Hash Tag 是原子性的物理边界**
```
Redis Cluster 的原子性边界 = Hash Slot（16384 个槽）
  ├── 单 Key 操作天然对应该 Key 所在 Slot
  └── Lua 多 Key 操作要求所有 Key 落在同一 Slot → 否则 CROSSSLOT

Hash Tag {xxx} 机制 → 仅 {xxx} 内文本参与 CRC16 计算
  示例: "{goods:1}:stock:bucket0" 与 "{goods:1}:record:user888"
    → 均 Hash 到 CRC16("goods:1") → 同一 Slot → Lua 原子执行 ✓

面试官追问："那所有 Key 用同一个 Tag 不就行了？"
  答: 可以，但等于放弃多商品分片隔离 → 所有流量回退到单节点。
  折衷方案: 按 {goodsId} Tag → 同商品内原子 ✓ + 异商品间隔离 ✓
```

**原则二：业务路由必须相容于业务目标**
```
路由策略          │ 负载均匀 │ 捡漏可行 │ 适用阶段
─────────────────┼─────────┼─────────┼──────────
一致性 Hash(user) │    ✓     │    ✗    │ 状态粘滞场景（WebSocket 长连）
随机路由          │    ✓     │    ✓    │ 无状态高并发 + 重试捡漏
轮询              │    ✓     │    ✓    │ 同随机，但可预测（易被脚本利用）

本项目选择: 随机路由
  理由: 不绑定用户 → 每次重试都可能是捡漏机会 → 路由服务于业务目标
```

**原则三：分桶 ≠ 分片，两层维度不可混淆**
```
Redis Cluster 分片:   Key → Hash Tag → Slot → Node（物理机器隔离）
应用层分桶 (Bucket):  Key → bucket{N}（逻辑拆分，同 Slot 内多个 Key）

单商品场景:
  {goods:1}:stock:bucket0  ─┐
  {goods:1}:stock:bucket1  ─┤ 同一 Slot → 同一 Node
  ...                       │ → Lua 原子跨 Key ✓
  {goods:1}:record:user888 ─┘ → 分桶分摊 Key 级热点 ✓

多商品场景:
  {goods:1}:* → Slot A → Node 0 (手机流量不冲击台灯流量 ✓)
  {goods:2}:* → Slot B → Node 1
```

**面试中的"自曝缺陷"策略**：
```
当面试官不主动提 CROSSSLOT 时，你主动引出：
  "我一开始的设计也犯过一个错误——把库存 Key 和用户记录 Key
   放在同一个 Lua 脚本里，但没考虑 Redis Cluster 的 CROSSSLOT 限制。
   后来通过 Hash Tag 修正了，代价是单商品的所有 Key 集中在同一节点，
   分桶只能在同节点内分摊热点，无法实现跨节点水平扩展。
   如果单商品 QPS 再上一个量级，下一步我会考虑客户端本地预减库存。"

为什么这会加分：
  1. 证明你真正部署过 Redis Cluster（不是单机 demo）
  2. 展示了你对方案物理边界的清醒认知
  3. 给出了可演进的下一步方向
```

## 场景 3：下单后的支付时效与库存回收

### 3.1 限时支付锁定
- **面试重要性**：★★★★☆（高）
- **理由**：引出订单状态机设计、分布式锁与锁超时、以及"Saga 补偿"模式。面试官会关注状态流转的完整性和异常处理。
- **核心技术**：订单状态机（Spring State Machine / 枚举驱动）、Redis 分布式锁（Redisson）
- **数据流转**：
  ```
  订单状态流转:
    [抢购成功] → 待支付(PENDING_PAY) → [支付成功] → 已完成(COMPLETED)
                     │
                     └── [超时5分钟 / 主动取消] → 已取消(CANCELLED) → 库存回池
  
  Redis 锁生命周期:
    下单 → SET seckill:lock:{goodsId}:{userId} EX 300 (5分钟锁)
    支付 → 成功 → 锁自然保留 → 订单完成
    超时 → 锁过期删除 → 触发库存回池
  ```

### 3.2 静默回池与"捡漏"
- **面试重要性**：★★★☆☆ → 升为 ★★★★☆（涉及路由拓扑与业务逻辑相容性，架构级问题）
- **理由**：表面是业务流程流转，深层考验"路由策略与业务目标的相容性"。面试官极易追问："你用一致性 Hash，那用户被钉死在售罄桶上，怎么捡漏？"
- **核心技术**：Redis INCRBY（库存回加）、随机路由策略（见 2.2）、异步消息驱动
- **捡漏机制的物理可行性证明**：
  ```
  关键设计依赖：2.2 节采用随机路由而非一致性 Hash

  若用一致性 Hash（错误方案）:
    User-A 被 Hash 固定到 Bucket-0
    Bucket-0 售罄 → User-A 永远路由到空桶
    Bucket-1 超时回收 1 件 → User-A 物理上触碰不到 → 捡漏死局 ❌

  若用随机路由（正确方案）:
    User-A 每次重试 → ThreadLocalRandom.next(0,10) → 随机选桶
    第 1 次: Bucket-3 → 空
    第 2 次: Bucket-7 → 空
    第 3 次: Bucket-1 → 刚好撞上回收的 1 件 → 捡漏成功 ✓
    概率: 1/10 每请求，用户持续重试 → 100% 最终命中
  ```
- **数据流转**：
  ```
  订单超时 → 消费者收到延时消息 → 核验支付状态(未支付)
    → INCRBY {goodsId}:stock:bucket{N} 1  (库存 +1，回池至原桶)
    → UPDATE order SET status='CANCELLED'

  此时仍在反复重试的用户:
    → 随机路由到某桶 → Lua 脚本检查该桶库存
    → stock 可能为 0 (未命中回收桶) 或 1 (命中回收桶)
    → 命中 → 捡漏成功
    → 未命中 → 返回失败，用户继续重试 → 重新随机 → 可能命中
  ```

### 3.3 超时自动释放
- **面试重要性**：★★★★☆（高）
- **理由**：引出消息队列的核心使用场景——延时消息、死信队列、最终一致性。面试官会追问"为什么不用定时任务扫表？"以及"消息丢失怎么处理？"
- **核心技术**：RocketMQ 延时消息（delayLevel）、死信队列 (DLQ)、异步消费者
- **数据流转**：
  ```
  下单成功:
    1. 扣库存 (Lua 原子)    ← 同步完成
    2. 投递延时消息          ← 异步完成
       └── RocketMQ: send(msg, delayLevel=5)  // 5分钟延时
  
  5分钟后:
    RocketMQ 消息可见 → Consumer 拉取
      → 查询 MySQL: SELECT status FROM order WHERE id=?
        ├── status='PAID' → ACK, 什么也不做
        └── status='PENDING' → 回滚:
              ├── Redis INCRBY 库存回池
              ├── UPDATE order SET status='CANCELLED'
              └── ACK
  
  为什么不用定时扫表？
    100万订单 × 每秒扫描 = 数据库瞬时崩溃
    延时消息: 到期时才处理，零扫表成本
  ```

## 场景 4：恶意流量与反欺诈

### 4.1 高频访问自动拦截
- **面试重要性**：★★★★☆（高）
- **理由**：网关层限流是后端工程师的必备技能。面试官会考察算法理解（令牌桶 vs 漏桶 vs 滑动窗口）、Redis 实现细节、以及熔断降级策略。
- **核心技术**：Redis Lua 滑动窗口限流、Guava RateLimiter、Sentinel / Hystrix
- **数据流转**：
  ```
  滑动窗口限流算法 (Redis Lua):
    key = "rate:{userId}"  窗口 = 1秒, 阈值 = 5次
    
    请求到达 → Lua:
      1. ZREMRANGEBYSCORE key 0 (now - 1000ms)   -- 清理过期记录
      2. ZCARD key                                -- 统计窗口内请求数
      3. if count >= 5 → return DENY
      4. ZADD key now now                         -- 记录本次请求
      5. EXPIRE key 1                             -- 续期1秒
      6. return ALLOW
  
  拦截发生在网关层（Spring Cloud Gateway Filter）:
    请求 → RateLimitFilter → 超频 → 429 Too Many Requests
                         → 正常 → 继续路由至秒杀核心
  ```

### 4.2 流量水位自适应验证
- **面试重要性**：★★★★☆（高）
- **理由**：这是"优雅降级"的经典案例，面试官会关注系统的弹性设计。与一刀切限流的区别在于：它在拦截脚本的同时保留了真人通道，是业务价值与技术方案的巧妙结合。
- **核心技术**：Sentinel 系统负载规则、验证码服务（算术题/滑块）、动态阈值配置
- **数据流转**：
  ```
  系统负载水位判断:
    当前 QPS < 10000  → 正常模式 → 直接下发抢购 Token
    当前 QPS ≥ 10000  → 降级模式 → 返回 Code: 4001 (需要验证)
    
  降级模式下的请求流转:
    用户请求 → Gateway
      → 检查当前 QPS (Redis 实时计数器)
      → QPS 超阈值 → 返回 {code: 4001, challengeId: "xxx"}
      → 前端弹出验证码 (算术题: "3+7=?")
      → 用户答题 (耗时 1~3秒) // 这1~3秒就是天然的削峰窗口
      → 前端携带 {challengeId, answer, userId} 请求验证
      → 验证通过 → 下发 Token → 继续抢购
  
  削峰效果:
    0ms:  10000 请求同时到达 (峰值)
    1s:   3000 人在答验证码, 7000 人直接进入
    2s:   2000 人陆续验证通过, 请求自然分散在 1~3 秒时间窗口
  ```

### 4.3 验证交互的无感融合
- **面试重要性**：★★★☆☆（中）
- **理由**：偏向前端交互设计，但后端的"请求上下文保持"和"幂等性保证"是面试中可引申的技术点。
- **核心技术**：前端 Axios 拦截器 + 请求重放队列、后端幂等性 Key (requestId)
- **数据流转**：
  ```
  用户视角: 点击 [立即抢购] 按钮 1 次
  系统视角:
    1. 请求 /api/seckill → 返回 4001 (需验证)
    2. Axios 拦截器捕获 4001 → 保存原始请求上下文 → 弹出验证组件
    3. 用户完成验证 → /api/verify-captcha → 成功
    4. 拦截器自动取回上下文 → 重放 /api/seckill (携带验证凭证)
    5. Controller 校验凭证 + 幂等性 Key → 执行抢购逻辑
  
  幂等性保证:
    每次点击生成唯一 requestId
    若重放 → 检查 requestId 是否已处理 → 是 → 直接返回已有结果
  ```
